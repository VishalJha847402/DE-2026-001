1
The WHY

Before we touch any code — why does this even exist?

Every topic in this course starts with three simple questions. If you have answers to these, you have something to hold on to while learning.

What is a variable? A variable is just a name that you give to a piece of data so you can use it again later. When you write name = "Vishal", you are telling Python: "Remember this text "Vishal" and call it name from now on."

What is a data type? A data type is a label that tells Python what kind of data you are storing. Is it a number? A piece of text? A list of items? Each type behaves differently — numbers can be added, text can be joined, lists can be looped over.

Why does this exist? Computer memory is just a long line of numbers. Without variables, you would have to remember exact memory addresses (like 0x7ffee5c1a4b0) for every piece of data. Without types, Python would not know if you mean to add two numbers or join two pieces of text. Variables solve naming. Types solve behavior.

Where will you use this in real DE work? Every single line of code in your future DE life touches a variable and a type. Some examples from real pipelines:

💼
Where this hits in DE • You read a CSV file → it becomes a list of dict objects.
• You hit an API → the response is a dict with mixed types inside.
• You query Postgres → numbers come back as int or float, dates come as str.
• You build a Spark job → every column has a type, and getting it wrong breaks the job at 3am in production.

So this is not a "beginner topic" you can skip. This is the topic on which Pandas, Spark, SQL connectors, and every API integration sit. Get this right, and the next 13 topics become easier. Get it wrong, and you will keep hitting strange bugs forever.

2
Variables — deep dive

A variable is not a box. It is a sticky note.

Most courses teach you variables as "boxes that hold values." That picture is wrong, and it will confuse you later. Let's build the right picture from day one.

The wrong picture (most courses teach this): "A variable is a box. You put 5 inside the box called x." This sounds nice, but it breaks the moment you do y = x. Did Python copy the 5 into a new box? No. Both x and y end up pointing to the same 5.

The right picture: A variable is a sticky note with a name on it. The actual data is an object sitting somewhere in memory. The sticky note points at the object. You can move the sticky note to a different object any time, but the object does not change because of that.

This sticky-note model explains every weird Python behaviour you will ever see — mutability bugs, function side effects, deep-vs-shallow copy, is vs ==. All of it.

🔑
The core mental model A variable is a label (sticky note) that points to an object (data sitting in memory). The label is cheap. The object is the real thing.

Let's see this in real Python code. We'll do something simple: create a variable, copy it, change the first one, and watch what happens to the second one.

Data Types — the 9 you must know

What kind of thing is this data?

Python has many types, but in real DE work, you will see these 9 over and over. Each one behaves differently. Each one solves a different problem.

When you create a variable, Python automatically figures out the type. You never have to tell Python "this is an int" or "this is a string" — it looks at what you wrote and decides. This is called dynamic typing. It is easier to write than languages like Java, but it also means you must be careful — Python won't catch a wrong type until you actually run the code.

The 9 types below split into two groups: immutable (cannot be changed after creation) and mutable (can be changed). We'll go deep on that in the next section.

int
Immutable
Whole numbers — positive, negative, or zero. No decimal point.
age = 28 order_id = 7821
float
Immutable
Decimal numbers. Even 3.0 is a float, not int.
price = 1499.50 gst = 0.18
str
Immutable
Text. Wrapped in single or double quotes. Both work.
name = "Anjali" city = 'Mumbai'
bool
Immutable
Only two values: True or False. Capital T, capital F.
is_active = True is_premium = False
None
Immutable
Means "nothing yet." Used for missing values, defaults.
last_login = None result = None
list
Mutable
Ordered collection. Square brackets. Can grow, shrink, change.
items = ["shoes", "socks"] prices = [299, 99]
tuple
Immutable
Like a list but cannot change. Used for fixed records.
location = (19.07, 72.87) rgb = (255, 0, 0)
dict
Mutable
Key → value pairs. Like JSON. The workhorse of DE.
user = { "id": 1042, "name": "Ravi" }
set
Mutable
Unique values only. No duplicates. No order.
tags = {"vip", "verified"} unique_ids = {1, 2, 3}

4
Mutability — the silent bug source

Can this object be changed after it's born?

This is the single most misunderstood idea in Python — and the source of most "weird" bugs in DE pipelines. Once you get it, you stop hitting these bugs forever.

Mutable means: the object itself can be changed after creation. You can add, remove, or update its contents without creating a new object.

Immutable means: once the object exists, it can never be changed. If you "change" it, Python actually makes a brand-new object and re-points your label to that.

Here's the key insight: when you pass a mutable object to a function, that function can change your original data — because both the caller and the function are pointing to the same object. When you pass an immutable object, no such danger exists.

Mutable — can change

list · dict · set

Same object stays. Contents change in place.

  • my_list.append(99) — list grows
  • my_dict["key"] = 10 — dict updates
  • my_set.add(5) — set grows
Immutable — cannot change

int · float · str · bool · tuple · None

To "change" it, Python makes a new object.

  • x = x + 1 — new int created
  • s = s + "!" — new string created
  • t = (1, 2) — cannot do t[0] = 9

Let's see the real-world impact of mutability with a tiny DE example. Imagine you wrote a helper function to add a "cleaned" flag to a list of transactions. A teammate calls your function — and complains that their original list got modified.

Type behavior — why "5" + 5 crashes

Why types decide everything Python does next

Same symbol, different meaning. The + sign means "add" for numbers but "join" for strings. Python decides which behavior based on the types involved.

Operators are type-aware. The + between two ints adds them. The + between two strings joins them. The + between a string and an int? Python doesn't guess — it crashes with a TypeError. Python forces you to decide what you meant.

This strict behavior is actually a good thing. It means data type mistakes get caught loudly, not silently. Imagine if Python auto-converted "5" + 5 to "55" — your financial pipeline would silently treat string IDs as numbers and corrupt millions of records. Python refuses to do that. It crashes, you fix it, the data stays clean.

🐍type_behavior.py
123456789101112

It raises TypeError. print("5" + 3) # 💥 TypeError # You must explicitly convert. Choose your meaning. print(int("5") + 3) # str → int → 8 print("5" + str(3)) # int → str → "53" # f-strings auto-convert — modern, clean, recommended user_id = 1042 print(f"User #{user_id} loaded.") # → "User #1042 loaded."
OUTPUT8 53 TypeError: can only concatenate str (not "int") to str 8 53 User #1042 loaded.
Line by line — every line, explained
Line 2
int + int does math. Both operands are numbers, so + means addition. Result: 8.
Line 3
str + str does concatenation. Both operands are text, so + joins them. Result: the 2-character string "53". Note: not the number 53.
Line 6
str + int crashes. Python sees two different types and refuses to guess your meaning. The error message tells you exactly what went wrong: can only concatenate str (not "int") to str.
Line 9
Explicit cast to int. int("5") converts the string "5" into the number 5. Then + 3 works as math.
Line 10
Explicit cast to str. str(3) turns the number into the text "3". Then "5" + "3" is joining → "53".
Line 13-14
f-string is the clean modern way. Anything inside {...} gets converted to a string automatically. This is what you'll use 95% of the time in real code.

6B
datetime — the DE timestamp toolkit

Every pipeline timestamps every row. Master this or hit bugs daily.

Python has 4 datetime types: date, time, datetime, timedelta. Most DE work uses datetime and timedelta.

  • datetime.now() — current local time. datetime.utcnow() — UTC.
  • ISO format: dt.isoformat() → "2026-06-13T10:30:00". Parse: datetime.fromisoformat(s).
  • Format strings: dt.strftime("%Y-%m-%d") format → string. strptime parse string → datetime.
  • Arithmetic: dt + timedelta(days=1). Subtract two datetimes → timedelta.
  • Timezone aware vs naive: Always use timezone-aware in production. datetime.now(timezone.utc).
🐍datetime_de.py
12345678910111213
from datetime import datetime, date, timedelta, timezone now = datetime.now(timezone.utc) # aware print(now.isoformat()) # 2026-06-13T10:30:00+00:00 # Parse API timestamp api_ts = datetime.fromisoformat("2026-06-13T10:30:00") # Last 7 days range for backfill today = date.today() week_ago = today - timedelta(days=7) print(f"backfill from {week_ago} to {today}") # Format for partition path print(now.strftime("year=%Y/month=%m/day=%d")) # Hive partition
⚠️
The naive vs aware bug datetime.now() gives naive (no tz info). datetime.utcnow() also naive. Mixing naive + aware raises TypeError. Production rule: always pass tz=timezone.utc. Store UTC, convert at display.

Things most beginners get wrong

Break it — edge cases that bite in production

What surprises even experienced developers

Four edge cases. Memorize them. You will hit each one in real DE work within your first year.

🐍surprises.py
1234567891011121314
# 1. bool is actually a subclass of int. True is 1, False is 0. print(True + True) # → 2 # 2. Empty containers and zero are FALSY in if-statements. if not []: print("empty list is False") if not "": print("empty string is False") if not 0: print("zero is False") # 3. Float math is NOT exact. This is where billing bugs come from. print(0.1 + 0.2) # → 0.30000000000000004 # 4. == checks values. is checks object identity. Not the same! a = [1, 2] b = [1, 2] print(a == b, a is b) # → True False
Why each edge case matters in real DE work
#1 bool
bool inherits from int. If you count truthy values with sum(my_bools), Python silently treats True as 1. Useful sometimes — confusing if you don't know.
#2 falsy
Empty containers are falsy. An if df: check on an empty DataFrame, an empty API response, or a zero count can silently skip your logic. Always check explicitly: if df.empty or if len(items) == 0.
#3 floats
Floats are not exact. 0.1 + 0.2 isn't 0.3 exactly. For money or financial calcs, use Decimal from the decimal module — never raw floats.
#4 is vs ==
Use == for value comparison. Use is only for is None or is not None. Using is elsewhere is a common bug.

The WHY

Why control flow exists — the problem it solves

Without control flow, your program would only run top to bottom, line by line, once. No decisions. No repetition. No real work done.

  • What is control flow? Rules that decide which lines run, in what order, and how many times.It's the "brain" of every program. Pure logic without any data work yet.
  • Why does it exist? Real-world problems need decisions and repetition. "If customer is premium, give 20% off." "For every row in the CSV, clean the date column." You can't write either without control flow.
  • Where in real DE work?
    • Every if in a pipeline that decides: backfill vs incremental load
    • Every for loop that processes rows, files, or API pages
    • Every while loop that retries a failed API call
    • Every list comprehension that transforms a column in one line

2
Decisions — if / elif / else

Branching — pick exactly one path based on a condition

The most basic decision tool. Python checks each condition top to bottom. The first one that is True wins. Everything else is skipped.

2.1The 4 key rules

  • Conditions are evaluated top-down. The first True wins. Order matters.
  • Only ONE branch runs. Even if 3 conditions are true, only the first one executes.
  • elif means "else if". You can chain as many as you need.
  • else is the fallback. Runs only if every if/elif above was False. Optional.

# Real DE rule: choose customer discount based on total spend total_spend = 45000 if total_spend >= 100000: discount = 0.20 # 20% — VIP tier elif total_spend >= 25000: discount = 0.10 # 10% — premium tier else: discount = 0.0 # 0% — regular print(f"Discount: {discount*100}%") # → Discount: 10.0%
OUTPUTDiscount: 10.0%

Customer's total spend = 45000. This is the value the decision depends on.
Line 4
First condition: is 45000 ≥ 100000? False. Skip this branch. Move to next.
Line 6

Line 8-9

2.2Comparison operators you'll use daily

  • == equal to Not = — that's assignment
  • != not equal to
  • > < greater / less
  • >= <= greater-equal / less-equal
  • and both must be True
  • or at least one must be True
  • not flips True ↔ False
  • in is value inside collection?
💥

2.3Indentation matters in Python

  • Python uses indentation to group code blocks. No curly braces like Java or C.
  • Standard is 4 spaces. VS Code auto-handles this.
  • Mixing tabs and spaces breaks code. Pick one. Stick with it.
  • The block ends when indent goes back to original level. No end keyword, no closing brace.
3
For loops — iterate over things

Run the same code for every item in a collection

If you have a list of 1000 customers and need to do something to each one, you use a for loop. Python takes one item at a time, runs your block, then moves to the next item.

3.1The mental model

  • "For each X in this collection, do Y." That's literally what the syntax reads as.
  • X is a temporary variable. It holds one item per round. Name it anything sensible (row, user, n).

  • Loop stops automatically. When no more items left, Python exits the loop.

While loops — repeat until something changes

When you don't know how many times to loop

for is for "I know I'm going through these items." while is for "keep going until this condition becomes False."

4.1When to use while vs for

  • Use for when you have a collection or fixed count (almost always your first choice).
  • Use while when you need to retry, poll, or wait for a state to change.
  • Danger: while can become an infinite loop if the condition never becomes False. Always have a clear exit.

Break it — edge cases that bite in production

Five surprises that catch even mid-level Python devs

  • 1. for-else exists, and it confuses everyone. The else on a for loop runs only if the loop finished WITHOUT a break. Used for "did I find what I was looking for?" Often clearer to use a flag variable instead.
  • 2. Mutating a list while iterating breaks Python silently. Items get skipped. Loop over list.copy() or build a new list with comprehension.
  • 3. Late binding in closures. A list of lambdas inside a loop all share the same loop variable — they all evaluate to the FINAL value, not what you'd expect. (Comes up in Topic 6+.)
  • 4. range() is lazy. range(10**10) doesn't use 10 GB of memory — it's a generator. Don't try to print it directly.
  • 5. True and False are int subclasses. sum([True, True, False]) gives 2. Use this trick to count truthy items in a list.

The WHY

Why OOP exists — the problem it solves

__init__ and self — the magic

The two things every beginner asks about

If you understand these two, you understand 70% of OOP. They're not magic — they're just Python keywords with very specific jobs.

3.1What is __init__?

  • __init__ is the constructor. Python calls it automatically the moment you write Customer("Ravi", "Mumbai").
  • You never call it directly. Calling the class name like a function Customer(...) triggers it for you.
  • Its job: set up the new object's initial data. Attach attributes via self.x = ...
  • The double underscores mean "Python-special." These are called dunder methods (double-underscore). We'll meet more of them in Section 7.
  • If you don't write __init__, Python provides a default empty one. Your object will exist but have no attributes.

3.2What is self?

  • self means "this specific object." When you're inside a method, self refers to whichever instance called the method.
  • Python passes self automatically. When you call c1.activate(), Python silently turns it into Customer.activate(c1) behind the scenes.
  • You don't pass self manually. Only Python does. You just include it as the first parameter of every method.
  • Why the word "self"? Pure convention. You could technically name it anything — but every Python codebase uses self. Stick with it.
⚠️
Forgetting self is the #1 OOP beginner bug If you write def activate(name): instead of def activate(self):, Python gives a confusing error: "activate() takes 1 positional argument but 2 were given." That's Python silently passing self and your method not expecting it.

Attributes — instance vs class

Two kinds of data lives on a class

Most attributes belong to one specific object (instance attributes). A few belong to the class itself, shared by every object made from it (class attributes). Knowing the difference prevents a famous category of bugs.

Instance attribute

Belongs to one object

Set inside __init__ via self.x = .... Different for each instance.

  • self.name
  • self.amount
  • Each object has its own copy
Class attribute

Shared by every instance

Set directly inside the class body, NOT in __init__. One single copy.

  • tax_rate = 0.18
  • company = "Acme"
  • All objects see the same value

Dunder methods — Python's special hooks

Customize how your objects behave with built-in operators

Dunder = "double underscore." These are methods Python calls automatically in certain situations. __init__ is one. There are many more — and using them makes your classes feel native.

  • __init__(self, ...) — called on object creation. You already know it.
  • __str__(self) — called by print() and str(). Returns a human-readable string.
  • __repr__(self) — called in REPL or by repr(). Returns a developer-debug string. Should look like Python code that could recreate the object.
  • __eq__(self, other) — called by ==. Define when two objects are equal.
  • __len__(self) — called by len(). Return a number.

Public · _protected · __private

Conventions for "you can use this" vs "stay out"

Python doesn't have true private attributes (unlike Java). It uses naming conventions — agreements between developers, not enforced rules. Three levels.

  • Public — no underscore: self.name. Anyone outside can read and modify it. The default.
  • Protected — one underscore: self._cache. Convention says: "internal to this class and its subclasses — don't touch from outside." Python doesn't enforce this. It's a polite hint.
  • Private — two underscores: self.__secret. Python actually renames the attribute behind the scenes (name mangling) to make accidental access harder. Still not truly private but stronger hint.
📌
What you'll see in real DE code Most projects use only public and _protected. Double-underscore is rare. The rule of thumb: anything starting with _ is internal — don't touch it unless you wrote it.

Things most beginners get wrong

❌ Wrong belief

__init__ creates the object.
No — Python creates the empty object first (via __new__), THEN calls __init__ to fill it. __init__ initializes, doesn't construct.
self is a Python keyword.
It's just a convention. You could name it anything. Don't. Every Python codebase uses self.
Class attributes are like instance attributes.
Class attributes are SHARED across all instances. If one is mutable (like a list) and you mutate it, every other instance sees the change. Major bug source.

In Python they're just hints. obj._cache still works from outside. The underscore says "don't, please" — it's not enforced.

Methods access data only via self.x. If you forget self, you get a different (local) variable that vanishes when the method exits.

Break it — edge cases that bite in production

Six surprises every OOP beginner hits

  • 1. Mutable default in __init__ — covered in the Hard practice. Always use None + check inside.
  • 2. Forgetting self in a method signature — Python gives a confusing "too many arguments" error.

Inheritance — extend, don't duplicate

One blueprint inherits from another

Inheritance is "child class IS-A parent class." A SavingsAccount is-a BankAccount with extras (interest). You don't rewrite balance, deposit, withdraw — you inherit them.

  • Syntax: class Child(Parent):
  • Child gets: all attributes + methods of parent automatically.
  • Child can: add new methods, override existing ones, or both.
  • "IS-A" test: "Is a SavingsAccount a BankAccount?" Yes → inheritance fits. "Is a Car a Engine?" No → don't inherit, use composition instead.

super() — call the parent

Extend the parent's behavior without losing it

Most child __init__ methods do two things: set up child-specific data AND ask the parent to do its setup. super() is how you call the parent.

  • super().__init__(...) calls the parent's __init__.
  • Use it at the START of child __init__ usually — parent does its setup, then you add child stuff on top.
  • Used in every Airflow custom operator: super().__init__(task_id=task_id, **kwargs)
  • Also works for any inherited method: super().run(), super().save().
🐍super_demo.py
12345678910111213
class BankAccount: def __init__(self, holder): self.holder = holder self.balance = 0 class SavingsAccount(BankAccount): def __init__(self, holder, rate): super().__init__(holder) # parent sets holder + balance self.rate = rate # child adds rate s = SavingsAccount("Ravi", 0.04) print(s.holder, s.balance, s.rate) # → Ravi 0 0.04

Forgetting super().__init__() is a famous bug If the child has its own __init__ and doesn't call super().__init__(), parent setup is skipped. Attributes you expected to exist won't. Always call super().__init__() in child __init__ unless you have a clear reason not to.

Multiple Inheritance & MRO

When a class has more than one parent — which one wins?

Python allows a class to inherit from multiple parents. Order matters. When methods clash, Python uses MRO (Method Resolution Order) to decide.

  • Syntax: class C(A, B):
  • MRO rule: left to right, depth-first, but using C3 linearization (Python's specific algorithm).
  • Check MRO: C.__mro__ or C.mro().
  • Mixin pattern: a small focused class designed to add one capability via multiple inheritance. class MyOp(LoggingMixin, BaseOperator): — common in Airflow.
  • Rule of thumb: if your MRO is hard to reason about, you're misusing inheritance. Prefer composition.

Guarantee cleanup, even when code crashes

Every with statement in Python uses a context manager. with open(...) as f: closes the file no matter what. You can build your own for DB connections, Spark sessions, transactions, timers.

  • __exit__(self, exc_type, exc_val, exc_tb) — runs when with block ends — even on error.
  • Used for: DB connection open/close, transaction begin/commit/rollback, file open/close, timers, locks, Spark session lifecycle.
  • If your code throws inside with — __exit__ still runs. Cleanup is guaranteed.

Composition vs Inheritance — senior wisdom

"Favor composition over inheritance" — the most-quoted OOP rule

Inheritance models "IS-A" (SavingsAccount IS-A BankAccount). Composition models "HAS-A" (Pipeline HAS-A Logger). When unsure, prefer composition.

  • Inheritance traps: deep hierarchies become fragile. Change a parent → breaks 20 children. Hard to override partially.
  • Composition wins: small focused objects + plug them together. Easier to test (mock the dependency). Easier to swap implementations.
  • Real DE example: Pipeline doesn't INHERIT from Logger — it HAS-A logger you pass in. Swap to a fake logger for testing.

Rule of thumb • IS-A relationship + truly shares behavior → inheritance.
• HAS-A relationship or just reusing logic → composition.
• If you'd want to swap the dependency in tests → composition. Always.

If the child has its own __init__, always call super unless you have explicit reason. Parent setup gets skipped otherwise.
ABC is just a convention.
ABC actually prevents instantiation. Forgetting to implement an abstract method gives TypeError immediately, not later.
Multiple inheritance is always bad.
Multi-inheritance with mixins (small focused classes) is fine and common. Multi-inheritance with two big parents fighting over the same methods is bad.
Context manager is only for files.
Context managers work for any resource — DB connections, transactions, locks, timers, Spark sessions, even temporary directories.
Overriding a method replaces the parent's logic.
It replaces by default. Call super().method() to also run the parent's logic when you want both.

Break it — edge cases that bite

Five Pydantic surprises in production

  • 1. Coercion accepts "yes" as bool True. Pydantic is lenient by default. Use ConfigDict(strict=True) for paranoid validation.
  • 2. Optional ≠ default. x: int | None means "must be int or None — required." Use x: int | None = None for actually optional.
  • 3. Mutable defaults — same trap as classes. Use Field(default_factory=list) for list/dict defaults.
  • 4. Validators run BEFORE coercion by default. Use @field_validator("x", mode="before") for raw input, mode="after" (default) for typed value.
  • 5. JSON keys with hyphens. {"x-rate": 1} can't map to Python x-rate. Use aliases.

The WHY — async

Why async exists — the I/O bottleneck

A DE pipeline spends 95% of its time waiting — for DB queries, API responses, S3 downloads. Sync code waits one at a time. Async code waits for all of them at once. Same machine, 10-100x faster.

  • Sync world: "Call API → wait 200ms → next call → wait 200ms..." 100 calls = 20 seconds.
  • Async world: "Fire all 100 calls → wait for whichever returns first → handle them as they finish." 100 calls = ~250ms total.
  • Why this matters in DE: ingesting from 1000 API endpoints, backfilling 10 years of data, batch-loading thousands of records — async makes the difference between 30 minutes and 30 hours.
  • Where in real DE work?
    • Async DB drivers — asyncpg for Postgres, aioboto3 for S3
    • Async HTTP clients — httpx.AsyncClient
    • FastAPI endpoints — all async by default
    • Modern Airflow async sensors
    • LangChain async chains, agent calls
    • Streaming ingestion — Kafka async consumers
⚡
The mental model Async = "I'm waiting for I/O. While I wait, run someone else's code." The event loop juggles thousands of "waiting" tasks on a single thread. No threading complexity, no GIL battles.

async / await basics

Two new keywords. That's mostly it.

A function marked async def is a coroutine — not run immediately, returns a "promise to run." You await it to get the result.

  • async def declares a coroutine function. Calling it returns a coroutine object, NOT the result.
  • await pauses the current coroutine until the awaited one finishes. You can only use await inside an async def.
  • asyncio.run(main()) starts the event loop and runs your top-level coroutine. Used once at entry point.
  • You can't mix: can't await a sync function. Can't call async function from sync code without asyncio.run().

The WHY

Why picking the right structure decides your pipeline speed

"Is user_id in this list of 10 million?" Done as a list: 10 million comparisons. Done as a set: 1 lookup. Same problem, 10 million times slower with wrong structure.

  • Why data structures exist: different problems need different tradeoffs. Ordered or not? Mutable or fixed? Fast lookup or fast append? Each structure picks a tradeoff.
  • Where in DE work?
    • Lists — every batch of rows, every result set, every API page
    • Dicts — every JSON record, every row of a DataFrame, every config
    • Sets — finding unique users, deduplicating, set operations between two days of data
    • Tuples — composite keys for dicts, returning multiple values, schema definitions
    • Counter — top-N frequencies (top failed jobs, most active users)
    • defaultdict — group-by pattern: {status: [order_ids]}

List — ordered, mutable, indexed

The workhorse of Python collections

  • When to use: ordered sequence, you'll add/remove items, you access by position.
  • Big O: append O(1), insert at front O(n), lookup by index O(1), in check O(n).
  • Methods: append, extend, insert, remove, pop, sort, reverse, index, count.
  • Slicing: lst[start:stop:step] — non-destructive, returns new list.

Tuple — ordered, immutable, hashable

The frozen list — use when nothing changes

  • When to use: fixed-shape record (coordinates, RGB, date+time pair). Multiple return values from a function. Hashable composite keys for dicts/sets.

  • Unpacking: a, b, c = my_tuple. Star: first, *rest = items.
  • Named tuples (preview): from collections import namedtuple — tuple with attribute access. Modern: use @dataclass(frozen=True) instead.

Dict — the most important structure in DE

Every JSON record, every row, every config — it's all dicts

  • When to use: key → value lookup, O(1) access. Configs, records, indexes, lookup tables.
  • Big O: insert O(1), lookup O(1), in check O(1).

Scope — the LEGB rule

Where Python finds your variable

When you use a name inside a function, Python looks in 4 places, in order: Local → Enclosing → Global → Built-in. First hit wins.

  • Enclosing: in a containing function (when nested).
  • Global: at module top level.
  • Built-in: from Python itself — print, len, sum.
  • Assignment creates Local by default. Use global x or nonlocal x to write outer scope.

Decorators — wrapping functions

The single most important pattern in modern Python

A decorator is a function that takes another function and returns a new one with extra behavior. @retry, @pytest.fixture, @field_validator, @app.get — all decorators.

  • Mental model: @dec / def f(...) is sugar for f = dec(f).
  • Pattern: outer function takes fn, inner wrapper does pre/post work + calls fn, returns wrapper.
  • Always use @functools.wraps(fn) on the wrapper — preserves the original function's name, docstring, signature.

import functools, time def timed(fn): @functools.wraps(fn) def wrapper(*args, **kwargs): start = time.time() result = fn(*args, **kwargs) print(f"{fn.__name__} took {time.time() - start:.3f}s") return result return wrapper @timed def run_etl(): time.sleep(0.2) run_etl() # run_etl took 0.20s

Higher-order functions — pass + return functions
  • map(fn, iterable) — apply fn to each item.
  • filter(fn, iterable) — keep items where fn returns truthy.
  • sorted(iter, key=fn) — sort by fn output.
  • Pythonic preference: list comprehensions are often clearer than map/filter. Use the form that reads best.

When you process millions of rows, hand-rolled loops are slow + memory-hungry. itertools gives you C-level fast iteration tools for batching, grouping, accumulating.

  • Why: regular slicing list[a:b] needs a list. islice works on generators, streams, infinite sequences.

Pure functions — the testability secret

A pure function: (1) same input → same output, (2) no side effects (no global writes, no I/O, no mutation of args). They're testable, cacheable, parallelizable, refactorable.

  • Pure: def add(a, b): return a + b — same input, same output, no side effects.
  • Impure: writes to DB, mutates a list, prints, reads now(), hits an API.
  • DE pattern: separate pure logic from I/O at boundaries. Mock the I/O, test the logic.
  • You can @lru_cache pure functions safely. Impure → cache lies.

Generators — yield for chunked iteration

The single most important DE feature in functions

A generator produces values lazily, one at a time, without loading everything into memory. Reading a 1TB file? Generators. Streaming API pages? Generators. This is how real DE handles big data on small RAM.

  • yield instead of return turns a function into a generator.
  • Returns a generator object. Iterate with for or next().
  • Lazy — produces one value per iteration. No memory blowup on huge data.
  • State is preserved between yields. Function pauses, resumes on next next().
  • Generator expressions: (x*2 for x in big_iterable) — same as list comprehension but lazy.
  • yield from delegates to another iterable. Used for chaining generators.

Common confusions
❌ Wrong belief
✅ Right understanding
Mutable default args are fine.
They are shared across calls — silent bug source. Use None sentinel.
return without value returns nothing.
It returns None. Same as a function with no return.
Decorators can only do simple wrapping.
Decorators with arguments need 3 levels (decorator factory). Class-based decorators also work.
Closure variables can be modified freely.
You need nonlocal to write to enclosing scope variables.
Lambdas are always faster than def.
Identical performance. Lambdas are just sugar for one-line def.
map/filter return lists.
They return iterators in Python 3. Wrap in list() if you need a list.

Break it — edge cases
  • 1. Mutable default trap — most famous Python bug. Always use None.
  • 2. Late binding in loop-created lambdas: [lambda: i for i in range(3)] — all return 2. Use lambda i=i: i to bind early.
  • 3. Forgetting functools.wraps in decorators — wrapped function loses its __name__ and docstring.
  • 4. nonlocal vs global: nonlocal writes to enclosing function, global writes to module. Easy to confuse.
  • 5. return inside a generator just stops iteration — value is on the StopIteration. Surprising.
